## Introduction

In <a href={link('/writing/running_linux_kernel_with_minimal_fs_in_bbb_part1')}>Part 1</a> we built the cross-compilation toolchain. This part is about booting: what the BeagleBone Black (BBB) expects on an SD card, how to build and place U-Boot, how I talk to the board over serial on macOS, and what a successful first boot looks like.

## What is a bootloader?

A bootloader is the first software executed by a hardware platform. Its job is to initialize enough hardware and load the next stage (usually a kernel).

The booting process is different from platform to platform but usually falls into one of two categories:

1. **No ROM help** — the processor does not assist with installing or loading a bootloader, you need detailed hardware knowledge (I am not sure how that path works in practice).
2. **ROM bootstrap** — the processor has ROM code that takes part in early boot and helps load the next stage.

The AM3358 SoC on the BBB falls into the second category thankfully.

## Booting from an SD card on the BBB

Unfortunately figuring out how your SoC boots is no easy task, you have to read a couple of technical docs from the board and SoC vendors. I’ll split this into: choosing SD boot, then what the ROM loads, then how the card is usually laid out.

### SD boot vs eMMC and the boot switch

On BBB by default it boots from eMMC (embedded Multi-Media Card), but we do not want that, we want to boot from an SD card that has our bootloader, kernel, and so on.

To change that on the BBB there is a boot switch on the board that we have to hold while it powers up, see [section 5.3.5 boot modes](https://docs.beagleboard.org/beaglebone-black.pdf).

### ROM code, SPL, `MLO`, and full U-Boot

Now we need to know what the SD card should look like and what it must contain. For that we read the SoC docs (AM3358) about how the ROM behaves.

I found this [AM335x U-Boot User's Guide](https://tha.de/~hhoegl/home/elinux/bbb/AM335x-U-Boot-User%27s-Guide.pdf) which you can read if you want more details, but basically it describes a two-stage U-Boot flow.

```quote
In AM335x the ROM code serves as the bootstrap loader, sometimes referred to as the Initial Program Loader (IPL)
or the Primary Program Loader (PPL). The booting is completed in two consecutive stages by U-Boot [1] binaries.
The binary for the 1st U-Boot stage is referred to as the Secondary Program Loader (SPL) or the MLO. The binary
for the 2nd U-Boot stage is simply referred to as U-Boot. SPL is a non-interactive loader and is a specially built
version of U-Boot. It is built concurrently when building U-Boot.
```

Why two stages? The first stage has to fit in static RAM, which is tiny, so you cannot fit a full-featured bootloader there. You ship a minimal loader that brings up basics (especially DDR RAM), then that loader pulls the full U-Boot into main memory. After that, full U-Boot can load the kernel and whatever else you need.

### Typical SD layout

The same guide describes a common two-partition picture:

- **1st partition:** FAT32 with `MLO` (small bootloader) and `u-boot.img` (full bootloader).
- **2nd partition:** ext4 root filesystem (and often the kernel). We will add that when we get there.

## Building U-Boot for the BBB

There are a couple of bootloaders out there, but we will use [U-Boot](https://github.com/u-boot/u-boot?tab=readme-ov-file) because it matches the SoC guide.

The U-Boot README has a `Building the Software` section. This is what I followed.

Fetching U-Boot:

```bash
git clone https://source.denx.de/u-boot/u-boot.git/
cd u-boot
```

Point the build at the cross toolchain from Part 1:

```bash
export CROSS_COMPILE=arm-linux-
```

U-Boot uses `menuconfig` similar to how crosstoo-ng is configured in Part1 and has many predefined configs for many boards.

```bash
dev@b7cc910aeffd:/workspace/u-boot$ ls configs/
10m50_defconfig                                  evb-rk3328_defconfig                         MPC8548CDS_legacy_defconfig             radxa-e25-rk3568_defconfig
3c120_defconfig                                  evb-rk3399_defconfig                         mscc_jr2_defconfig                      radxa-zero2_defconfig
A10-OLinuXino-Lime_defconfig                     evb-rk3568_defconfig                         mscc_luton_defconfig                    radxa-zero_defconfig
A10s-OLinuXino-M_defconfig                       evb-rk3588_defconfig                         mscc_ocelot_defconfig                   rcar3_salvator-x_defconfig
```

For the BBB I used `am335x_evm_defconfig`:

```bash
dev@b7cc910aeffd:/workspace/u-boot$ make am335x_evm_defconfig
#
# configuration written to .config
#
dev@b7cc910aeffd:/workspace/u-boot$ 
```

Optional tweaks via `menuconfig`:

```bash
make menuconfig 
```

- SPL Configuration Options → Support Networking → off (keeps the SPL smaller).

Build:

```bash
make DEVICE_TREE=am335x-boneblack
```

You should get `MLO` (SPL) and `u-boot.img` (full U-Boot). Pick the device tree that matches your board, we will talk more about device trees in the next part. For now, think of it as a structured description of the hardware.

## Partitioning the SD card on macOS

With those artifacts in hand, the next step is partitioning the SD card and copying files to the right place.

On Linux I would normally use `cfdisk` or `parted`, on macOS I used `diskutil` instead.

Find the disk (double check the identifier so you do not wipe the wrong drive):

```bash
diskutil list
```

For me it was `/dev/disk4`.

Unmount and clear the start of the device (partition table area):

```bash
diskutil unmountDisk /dev/disk4
```
```bash
sudo dd if=/dev/zero of=/dev/rdisk4 bs=1m count=16
```

Create a small FAT32 boot partition, leave the rest unallocated for later (rootfs, etc.):

```bash
diskutil partitionDisk /dev/disk4 MBRFormat "MS-DOS FAT32" boot 64M "Free Space" unused R
```

Mark the first partition as boatable by using `fdisk`:

- print
- flag 1
- print
- write
- y
- exit

## Serial console

Before plugging in power with the new card, it helps to have the serial console ready. this is where bootloader (and soon kernel) logs show up.

You might ask why use serial? why not other port like USB? as far as I understand serial port is very simple that it can send data in very early bring up (HW initialize very early). In contrast USB initialize way later on I think.

I use this [serial to USB adapter](https://amzn.eu/d/0d2KbItZ), wired to the serial debug header (pin order matters, match GND/TX/RX carefully, see Figure 2 for header placement on the board).

<Figure
  src={SerialSetup}
  alt="BBB serial setup"
  caption="Figure 4: BBB serial setup"
/>

`picocom` is used as serial communication program:

```bash
brew install picocom
```

List candidate devices, then open the right `cu.*` device at 115200 baud (idk what baud is but I see people use it):

```bash
ls /dev/cu.*
```
```bash
picocom -b 115200 /dev/cu.usbserial-UUT1
```

## First boot: our U-Boot over serial

With picocom connected, insert the SD card, hold the boot switch, and apply power so the board selects SD boot.

You should see the SPL hand off to U-Boot, the **build date** in the U-Boot banner should match when you compiled, that is a good sign you are running *your* images from the card, not an older copy from eMMC.
