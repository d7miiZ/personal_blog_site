## Introduction

I finished [Bootlin’s embedded Linux course](https://bootlin.com/training/embedded-linux/) and I decided to try and replicate what I learned on a real board. In this serries I will try to build/boot bootloader, kernel and a minimal fs in a BeagleBone Black (bbb) board, I got mine from [Amazon](https://amzn.eu/d/0ecv7bg8).

To build/boot the things that I mentioned before I would need to cross-compile all of it (bootloader/kernel/fs/etc), and for this I would need what we 
call a cross-compilation toolchain.

## Cross-compilation toolchain

Tools run on the **host** (e.g. `x86_64`), output for the **target** (here, 32-bit ARM).

Usually contains:

- **Cross compiler** (`gcc`)
- **Binutils** (`as`, `ld`, `objdump`, `strip`, …)—see [GNU Binutils](https://www.gnu.org/software/binutils/)
- **C library** (e.g. **glibc**, **uClibc**, **musl**)
- **Kernel headers** (aligned with the kernel you’ll run)
- **Debugger** (`gdb`, optional)

| How you get it | Tradeoff |
| ---------------- | -------- |
| Build everything yourself | Control, pain (too hard for me) |
| Prebuilt toolchain | Fast, rigid |
| Automated tools | Custom + educational |

I will do the last since it allows you to customize it + see what choices you need to make to build a cross-compilation toolchain.

## Install crosstool-NG

```bash
git clone https://github.com/crosstool-ng/crosstool-ng
cd crosstool-ng/
```

We first need to generate a configure script since crosstool-NG is build system uses [Autotools](https://www.gnu.org/software/automake/manual/html_node/Autotools-Introduction.html), once configure script is generated we pass to it `--enable-local` to keep installation locally and not system wide

```bash
./bootstrap
./configure --enable-local
make
```

If all is good you should be able to run this:

```bash
./ct-ng help
```

In [BBB high level spec](https://docs.beagleboard.org/boards/beaglebone/black/ch05.html) says the SoC (system on chip, which basically means CPU and other stuff close together in on chip, as far as I understand) is [Sitara AM3358BZCZ100](https://www.ti.com/product/AM3358/part-details/AM3358BZCZ100) which uses `Arm Cortex-A8` as CPU, so for me I will use the below as starting point (you choose what CPU your board use):

```bash
./ct-ng arm-cortexa9_neon-linux-gnueabihf
```

  Match **Target options** to **your** CPU, FPU, and ABI. Blind copy-paste → fun illegal instructions later.
</Callout>

```bash
./ct-ng menuconfig
```

### Target options

- **ARCH_FPU** → `vfpv3`
- **Floating point** → `hardware`

This tells Crosstool-NG to emit hardware FP instructions instead of software floating point, this entirely depends on what your CPU/target support

### Toolchain options

- **Tuple vendor** → `ash`
- **Tuple alias** → `arm-linux` → tools like `arm-linux-gcc`

To explain this we need to understand what are the tuples in toolchain name and what do they mean, example names:

#### Triplet

Examples: `arm-linux-gnueabihf`, `mips64el-linux-gnu`.

1. **Architecture** — `arm`, `riscv64`, …
2. **Vendor** (optional) — e.g. `ash`
3. **OS / env** — `linux`, `none`, …
4. **ABI / libc** — `gnueabihf`, `musleabihf`, …

### Operating system

- **Linux version** close to the kernel you’ll build (I aimed near **6.6**).

### C library

- **musl**, wanted to try it and lighter than **glibc** in theory.

### C compiler

- **GCC** → `14.3.0`
- **C++** enabled

Save and exit.

## Build

```bash
./ct-ng build
```

For me it took really long time (since Docker Linux in Mac OS), about 50 minutes :(, so go make tea or play a game (I recommend [Katana Zero](https://store.steampowered.com/app/460950/Katana_ZERO/))

Toolchain under `$HOME/x-tools/arm-*`. Add `bin` to `PATH`:

```bash
echo 'export PATH="/home/dev/x-tools/arm-ash-linux-musleabihf/bin/:$PATH"' >> ~/.bashrc
```

Side note: since we added `arm-linux` as alias in tuple setup above then we can call our toolchain like this `arm-linux-gcc`.


## Test The Toolchain

```c
#include <stdio.h>

int main(void) {
    printf("Hello, world!\n");
    return 0;
}
```

```bash
arm-linux-gcc -o hello hello.c
file hello
```

Expect 32-bit ARM ELF; **musl** shows up in the interpreter line, e.g.:

```text
hello: ELF 32-bit LSB executable, ARM, EABI5 version 1 (SYSV), dynamically linked, interpreter /lib/ld-musl-armhf.so.1, not stripped
```

## Clean

After building the cross-compilation toolchain we can now cleanup crosstool-NG stuff by running the following command:

```bash
./ct-ng clean
```
